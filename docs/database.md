# Схема Базы Данных

## Сущности MVP

### users

Хранит профиль пользователя Telegram.

| Поле | Тип | Назначение |
| --- | --- | --- |
| id | integer | Внутренний идентификатор |
| telegram_id | bigint | Идентификатор пользователя в Telegram |
| username | varchar | Username пользователя |
| full_name | varchar | Имя пользователя |
| source_language | varchar | Родной язык |
| target_language | varchar | Изучаемый язык |
| level | varchar | Уровень владения |
| created_at | datetime | Дата регистрации |
| updated_at | datetime | Дата обновления |

### word_sets

Готовые тематические наборы слов.

| Поле | Тип | Назначение |
| --- | --- | --- |
| id | integer | Идентификатор набора |
| title | varchar | Название темы |
| description | text | Описание набора |
| level | varchar | Рекомендуемый уровень |
| is_active | bool | Доступность в боте |

### words

Конкретные слова и фразы внутри набора.

| Поле | Тип | Назначение |
| --- | --- | --- |
| id | integer | Идентификатор слова |
| word_set_id | integer | Ссылка на набор |
| source_text | varchar | Слово на родном языке |
| target_text | varchar | Перевод |
| example | text | Пример употребления |

### training_attempts

История тренировок и результатов.

| Поле | Тип | Назначение |
| --- | --- | --- |
| id | integer | Идентификатор попытки |
| user_id | integer | Ссылка на пользователя |
| word_set_id | integer | Набор, по которому был тест |
| correct_answers | integer | Количество правильных ответов |
| total_questions | integer | Общее количество вопросов |
| created_at | datetime | Дата прохождения |

## Связи

```text
users 1 --- N training_attempts
word_sets 1 --- N words
word_sets 1 --- N training_attempts
```

## Обоснование

Этой схемы достаточно для MVP:

- есть профиль пользователя
- есть учебные наборы
- есть хранилище слов
- есть база для статистики и прогресса

На поздних этапах можно добавить таблицы пользовательских словарей, интервального повторения и событийной аналитики.

## Технологическое решение

В качестве основной БД выбран `PostgreSQL 16`.

Причины выбора:

- подходит для backend-проекта лучше, чем файловая БД
- одинаково работает у всех участников через Docker
- без переделки масштабируется на более сложную логику
- лучше выглядит на защите как промышленный стек
