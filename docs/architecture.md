# Архитектура

## Цель первого спринта

Подготовить технический фундамент для MVP Telegram-бота:

- определить стек
- зафиксировать структуру модулей
- описать модель данных
- согласовать основные диалоговые сценарии

## Архитектурный подход

Приложение строится как событийно-ориентированный backend, который принимает входящие обновления Telegram и переводит пользователя между состояниями интерфейса. На старте используется `long polling`, так как это проще для локальной разработки и первых показов. Инфраструктурная часть запускается через `Docker Compose`: отдельный контейнер приложения и отдельный контейнер PostgreSQL.

## Слои приложения

### 1. Handlers

Обрабатывают команды, нажатия кнопок и текст пользователя.

### 2. States

Описывают состояния пользователя в рамках `FSM`, например выбор языка или уровня.

### 3. Keyboards

Содержат Telegram-клавиатуры и навигационные элементы.

### 4. Services

Хранят бизнес-логику: регистрацию пользователя, получение прогресса, выдачу учебного контента.

### 5. Database

Содержит модели, engine, session factory и функции инициализации.

## Базовая схема модулей

```text
Telegram Update
    -> aiogram Dispatcher
        -> handlers
            -> services
                -> database
        -> keyboards
        -> states
```

## Решения по стеку

- `aiogram 3` — современный асинхронный фреймворк для Telegram-ботов
- `SQLAlchemy 2` — ORM и контроль схемы данных
- `PostgreSQL` — основная реляционная БД проекта
- `Docker Compose` — одинаковое окружение для всей команды
- `.env` + `pydantic-settings` — удобная конфигурация
