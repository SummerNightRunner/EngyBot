# Архитектура

## Обзор

EngyBot построен по простой архитектуре Telegram-бота:

- Telegram используется как клиентский интерфейс;
- `aiogram` маршрутизирует команды и callback-запросы;
- логика приложения реализована внутри сервиса бота;
- PostgreSQL хранит пользователей, контент, попытки и прогресс.

На текущем этапе проект остается монолитным. Отдельного frontend-приложения, REST API и admin service нет.

## Слои

### `handlers`

Точки входа для:

- `/start`;
- callback-навигации;
- онбординга;
- экранов словаря и grammar;
- квизов;
- review;
- progress-экранов.

Основная часть логики сейчас находится в `src/bot/handlers/start.py`.

### `keyboards`

Telegram inline keyboards для:

- главной навигации;
- меню course и practice;
- quiz actions;
- feedback-экранов;
- profile и progress-навигации.

### `services`

Вспомогательная логика приложения:

- загрузка seed-контента;
- фильтрация по CEFR-уровню;
- подбор слов для квизов и daily practice;
- форматирование учебных блоков.

### `states`

FSM-описания для сценариев, где нужен пошаговый маршрут:

- онбординг;
- training sessions;
- quiz flow.

### `database`

Слой хранения данных:

- SQLAlchemy models;
- async engine;
- session factory;
- инициализация схемы.

## Поток обработки запроса

Типовой маршрут:

```text
Telegram update
  -> aiogram router
    -> handler
      -> services / database
        -> state update
          -> response message + keyboard
```

Пример:

1. пользователь открывает `Daily practice`;
2. хендлер проверяет, существует ли уже сессия на текущую дату;
3. если сессия есть, она восстанавливается;
4. если сессии нет, бот создает новую и сохраняет ее;
5. после этого отрисовывается следующий вопрос.

## Почему выбрана такая схема

Текущая архитектура выбрана по практическим причинам:

- проект легко запускать локально и в Docker;
- отладка остается прозрачной;
- учебную логику можно развивать без лишних сервисов;
- схема данных остается управляемой.

Для текущего масштаба монолитный подход достаточен.

## Сильные стороны

- прогресс хранится постоянно;
- seed-контент отделен от interaction logic;
- daily practice хранится отдельно от runtime-state;
- контент фильтруется по уровню пользователя;
- bilingual UI встроен поверх существующей модели профиля.

## Узкие места

### Крупный handler-модуль

`src/bot/handlers/start.py` слишком большой и совмещает несколько зон ответственности:

- меню и навигация;
- course-экраны;
- quiz flow;
- progress;
- review.

Этот модуль нужно разбивать на feature-based файлы.

### Отсутствие unit-слоя

В проекте уже есть словарь, grammar, dialogues и practice. Но они пока не собраны в полноценную unit-based модель курса.

## Планируемая переработка

Следующие шаги:

1. разбить handlers по функциональным модулям;
2. ввести сущность `unit`;
3. добавить отдельные grammar practice flows;
4. вынести UI-строки в более явный слой локализации.
